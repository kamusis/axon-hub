# 云和恩墨报销明细生成器

## 工作原理

用户提供一个目录路径，目录中每张图片或 PDF 对应一条报销记录。Agent 遍历文件，识别票据内容，结合文件名推断用途，最终输出 CSV。

```
用户给目录路径 → Agent 扫描文件列表 → 逐文件提取内容 → 映射结构化字段 → 生成 CSV
```

核心思路：**文件名 = 上下文线索，文件内容 = 数据来源**。两者结合才能准确填充字段。

## 字段定义

读取 `references/field_definitions.md` 获取完整的字段定义、备选值列表和格式要求。

**字段速览（CSV 列顺序，共 14 列，与 `references/field_definitions.md` 完全对齐）：**

| # | CSV 列名 | 类型 | 说明 |
|---|-----------|------|------|
| 1 | 项目名称 | 自由录入 | 报销归属的项目名称 |
| 2 | 报销类型 | 备选值 | 费用大类 |
| 3 | 费用类型 | 备选值 | 费用细分类 |
| 4 | 费用发生起始日期 | 自由录入 | YYYY-MM-DD 格式 |
| 5 | 费用发生结束日期 | 自由录入 | YYYY-MM-DD 格式 |
| 6 | 报销金额 | 自由录入 | 纯数字，不含货币符号，保留两位小数 |
| 7 | 单据数 | 自由录入 | 通常为 1；补充单据不计入 |
| 8 | 费用说明 | 自由录入 | 简要说明费用用途 |
| 9 | 出差申请 | 自由录入 | **必须由用户提供，未提供时暂停询问** |
| 10 | 出差天数 | 自由录入 | 仅补助记录填写，其他记录留空 |
| 11 | 销售活动编号 | 自由录入 | 用户未提供时留空 |
| 12 | 活动类型 | 备选值 | 系统保留字段，暂无有效值，留空 |
| 13 | 报工单 | 自由录入 | 用户未提供时留空 |
| 14 | 外派申请 | 自由录入 | 用户未提供时留空 |

> **类型说明**：备选值型 = AI 从预定义列表中选择；自由录入型 = AI 根据文件内容生成建议文本。

## 工作流

### Step 1: 解析用户请求

从用户输入中提取：
- **目录路径**（必填）：如 `~/Documents/receipts/2026-05-出差` 或 `/Users/username/invoices/`
- **项目名称**（可选）：若未提供，使用 reference 中的默认值
- **出差申请编号**（**必填**）：若未提供，暂停处理并询问用户

**向用户确认时**，一次性列出所有缺失的关键字段，避免逐个追问。

### Step 2: 扫描目录

用 Bash 工具列出目录中所有支持的文件：

```bash
# Linux / macOS
ls -1 <目录路径>/*.{png,jpg,jpeg,bmp,pdf} 2>/dev/null
# 或跨平台方式（Python 脚本已内置文件扫描功能）
python scripts/extract_invoice_text.py <目录路径>  # 会列出所有处理的文件
```

**注意**：
- 仅处理根目录文件，不递归子目录（除非用户明确要求）
- 支持格式：`.png`, `.jpg`, `.jpeg`, `.bmp`, `.pdf`
- 过滤掉无关文件（如 `.DS_Store`, `Thumbs.db`）
- 按文件名自然排序

### Step 3: 逐文件提取信息

对每个文件执行以下操作：

#### 3.1 文件读取策略

**PDF 文件**（推荐方案，按优先级）：

1. **pdfplumber 提取文本**（电子发票 PDF，有文本层）
   - 运行 `scripts/extract_invoice_text.py <目录路径>`
   - 该脚本会自动用 pdfplumber 提取文本，失败则用 PyMuPDF 转图片
   - 依赖：`pip install -r scripts/requirements.txt`
   - 大多数电子发票（火车票、机票、住宿、餐饮）都有文本层，可直接提取

2. **PyMuPDF 转图片 + multimodal Read**（扫描件/图片 PDF）
   - 上述脚本自动将无法提取文本的 PDF 转为 PNG（200 DPI）
   - 输出到 `<目录>/_images/` 下
   - Agent 用 Read 工具读取生成的 PNG 图片

**图片文件**（`.png`, `.jpg`, `.jpeg`, `.bmp`）：

处理优先级：

1. **优先**：直接运行 `scripts/extract_invoice_text.py <目录路径>`
   - 该脚本已内置 JPG/BMP→PNG 转换逻辑，会对整个目录批量处理
   - PNG 文件会跳过转换，直接用 Read 工具读取
   - 这是首选方式，效率最高

2. **仅当脚本未覆盖的特殊情况**（如单独处理一个文件，且 Read 工具读取 PNG 时报错 `Cannot display content of binary file`），才需要 agent 手动执行以下 Python 代码将 JPG 转为 PNG：
   ```python
   from PIL import Image
   import os
   img = Image.open(jpg_path)
   img = img.convert('RGB')
   output_dir = os.path.join(os.path.dirname(jpg_path), '_images')
   os.makedirs(output_dir, exist_ok=True)
   png_path = os.path.join(output_dir, os.path.splitext(os.path.basename(jpg_path))[0] + '.png')
   img.save(png_path, 'PNG', dpi=(300, 300))
   ```
   转换完成后用 Read 工具读取生成的 PNG 文件。

> **注意**：上述 Python 代码片段是 fallback 方案，不是常规流程。绝大多数情况下只需运行 `scripts/extract_invoice_text.py` 即可，无需 agent 手动执行转换代码。

#### 3.2 提取关键信息

从文件内容中识别并提取：
- **金额**：发票金额、票价、总价等
- **日期**：开票日期、行程日期、入住/退房日期等
- **商家/服务方**：公司名称、航空公司、酒店名称等
- **发票号**：如有
- **明细**：如行程起止地、房型、服务内容等

**结合文件名**：文件名通常包含用途信息（如"南京住宿发票"、"北京-兰州机票"），与文件内容交叉验证。

**整理为结构化记录**：将提取的信息整理为一条记录的各字段值。

**提取策略**：
- 优先从文件内容（票据本身）提取金额和日期
- 文件名作为用途分类的辅助线索
- 如果文件内容模糊不清，标注 `[需人工确认]`

**批量处理建议**：对同一目录下的所有文件（PDF + 图片），优先运行 `scripts/extract_invoice_text.py` 一次性处理。脚本会自动对 PDF 提取文本或转图片，对 JPG/BMP 转为 PNG，PNG 文件则跳过（直接用 Read 读取）。比逐文件处理更高效。

### Step 4: 字段映射与填充

读取 `references/field_definitions.md`，对每个字段进行填充：

**4.1 行程时间线构建（必须先执行）**

在填充任何日期字段之前，先根据所有交通票据建立行程时间线：

1. **提取所有交通票据**（机票、火车票、汽车票），按日期排序
2. **确定每段行程**：根据出发地→目的地，确定旅客在每个城市的停留时间段
3. **输出时间线摘要**供后续校验：
   ```
   5/8: 上海→马鞍山（火车）| 名古屋→上海（飞机回程）
   5/11: 南京→汉口（火车）
   5/12-5/13: 武汉停留（住宿发票验证）
   5/13: 汉口→北京（火车）
   5/18: 天津→名古屋（飞机）
   ```

**4.2 日期合理性校验**

对于每张非交通类发票（餐费、住宿、打车等），执行以下校验：

- 发票上标注的**城市/商家**是否在某段行程的日期窗口内？
- 如果**城市匹配但开票日期不在行程窗口内**（常见于补开发票），则实际发生日期应推断为该城市行程期间
- 如果**城市不匹配任何行程段**，标注 `[需人工确认]`

> 这是考验 AI 推理能力的关键步骤。发票开票日期 ≠ 实际消费日期，必须结合行程逻辑判断。

**4.3 字段填充**

**备选值型字段**处理：
- 根据文件内容+文件名，从备选值列表中选择最匹配的值
- 如无法匹配，选择最接近的备选值并标注

**自由录入型字段**处理：
- 根据提取的信息生成简明准确的文本
- 遵循字段的格式要求（如日期格式、金额格式）
- **费用发生起始/结束日期**：使用校验后的实际发生日期，而非发票开票日期

**公共字段处理**（所有记录共享的字段）：
- **出差申请**：必须由用户提供，未提供时暂停处理并询问，不可默认填充
- **出差天数**：仅使用交通类票据（机票、火车票、大巴票等），取第一个交通票据日期到最后一个交通票据日期（含首尾），计算出差天数。排除明显与本次出差无关的交通票据（如出发前市内通勤）。票据不完整时询问用户
- **项目名称**：从 Step 1 的用户输入或 reference 默认值获取

**出差补助处理**（无发票、按天数计算）：

出差补助不来自任何票据文件，而是根据出差天数和政策自动计算。读取 `references/field_definitions.md` 中的"出差补助计算规则"章节获取详细标准。

1. **向用户确认**：首次处理时询问用户是否需要添加出差补助记录
2. **差旅补助**：出差天数 × 100元/天（含交通补助50元+误餐补助50元，港澳台地区标准相同），生成一条记录
3. **住宿补助**：（出差天数 - 1 - 有酒店住宿的晚数）× 100元/天，生成一条记录。其中"出差天数 - 1"是出差期间的总住宿晚数（返程日当天返回，不需住宿）。示例：出差11天，2晚有酒店发票 → 无酒店住宿晚数 = 11 - 1 - 2 = 8晚
4. **有酒店住宿晚数**：从住宿费发票的入住日期推断
5. **单据数**：补助记录无实体票据，填 `0`
6. **费用说明**：按 reference 中的示例格式填写

### Step 5: 生成 CSV

将所有记录写入 CSV 文件：

**CSV 规范**：详见 `references/field_definitions.md` 中的"CSV 输出规范"章节。

**输出路径**：
- 默认输出到用户指定的报销目录（与发票同目录）
- 文件名格式：`报销明细_{YYYY-MM-DD}.csv`

**Python 生成脚本**（推荐）：用 Python `csv` 模块写入，按 reference 中的 CSV 输出规范设置编码和格式。

### Step 6: 结果展示与确认

1. 使用 `preview_url` 或直接在回复中展示 CSV 内容预览（表格形式）
2. 标注以下信息：
   - 共处理了多少个文件
   - 哪些字段是 AI 自动填充的
   - 哪些字段标注了 `[需人工确认]` 及原因
   - 总报销金额汇总
3. 提醒用户检查并确认，特别是：
   - 金额是否准确
   - 费用类型是否正确
   - 费用说明是否通顺

## 质量检查清单

在输出 CSV 前，Agent 应自检：

- [ ] 每条记录的金额与发票上的金额一致
- [ ] 日期格式统一为 YYYY-MM-DD
- [ ] 备选值字段的值确实在备选列表中
- [ ] 费用说明简洁且能说明用途
- [ ] 同一出差的多条记录共享相同的项目名称和出差申请号
- [ ] 出差申请编号已由用户提供，未猜测默认值
- [ ] **费用发生日期经过行程合理性校验**：非交通类发票的日期与所在城市行程窗口一致
- [ ] CSV 编码和格式符合 `references/field_definitions.md` 规范
- [ ] **无重复计算**：检查生成的所有记录，如果发现金额+日期完全相同的两条记录，大概率是重复，必须追溯确认是否来自同一笔费用的多个文件（见下方"补充单据识别规则"）
- [ ] **补充单据已正确归并**：行程报销单、明细清单等补充性文件不计为独立报销记录，也不计入单据数（见下方"补充单据识别规则"）

### 补充单据识别规则

有些费用会伴随多份文件，但**只有一份是核心票据（发票），其余是补充说明**。Agent 必须识别并归并，避免重复计算。

**常见补充单据类型**：
- **行程报销单**：滴滴出行等平台会同时提供"电子发票"和"行程报销单"。电子发票是核心票据（有金额、税额），行程报销单只是补充说明（列出起止地点、时间，方便财务审计）。**只按电子发票金额生成记录，行程报销单不单独计费，也不计入单据数。**
- **明细清单**：酒店住宿的"消费明细单"是发票的补充，不单独记录。
- **订单截图**：机票/火车的订单确认截图，如果已有对应发票，则截图不单独记录。

**识别方法**：
1. **文件名配对**：同名不同后缀的文件（如"滴滴电子发票A.pdf"和"滴滴出行行程报销单A.pdf"）通常是对应关系
2. **金额交叉验证**：补充单据上的合计金额应与核心票据一致，如果不一致需标注 `[需人工确认]`
3. **提取有价值信息**：虽然补充单据不单独生成记录，但其中的起止地点、具体时间等信息可用于丰富费用说明（如"将台路→北京南站 滴滴出行"比"北京市内滴滴出行"更具体）

**归并原则**：一份费用 = 一条 CSV 记录，无论有几份配套文件。单据数只计核心票据（通常为 1）。

## 注意事项

1. **图片质量**：如果图片模糊、角度倾斜或裁切不全，在对应字段标注 `[需人工确认]`，不要猜测。
2. **金额精度**：报销金额保留两位小数，与发票面额完全一致。
3. **文件名解析**：中文文件名可能包含关键信息（城市名、费用类型），务必纳入分析。
4. **分组逻辑**：如果目录中的文件明显属于多次出差，应识别并提示用户确认是否需要分组。
5. **PDF 处理**：优先运行 `scripts/extract_invoice_text.py` 批量提取文本，扫描件自动转图片供 multimodal 读取。详见 Step 3.1。
6. **隐私安全**：提取的信息仅用于生成报销 CSV，不做任何其他用途。
